# Question 019: スキル不足による完成の定義の調整要求

## Scenario
開発チームが議論しています。開発チームメンバーの一人が不在のため、完成の定義（Definition of Done）に従って、動作する「完成」したインクリメントを提供するために必要なすべてのスキルをチームが持っていないことが判明しました。チームは完成の定義を調整したいと考えています。

## Question
スクラムマスターとして、この状況にどのように対応すべきでしょうか？

## Evaluation Criteria
1. **スクラムガイドの理解** - 完成の定義の目的と重要性、スキルの欠如に対する適切な対応の理解
2. **実践的な適用** - 完成の定義を調整する代わりに、スキル不足を解決する具体的な方法
3. **品質と透明性** - 完成の定義を緩めることの問題点と、代替的な解決策の提案
4. **学習目標達成** - 問題の学習目標に対する回答の完全性と明確性

## Model Answer

スクラムマスターとして、この状況は完成の定義の目的と、スキル不足に対する適切な対応に関する重要な問題です。完成の定義を緩めることは推奨されません。以下のアプローチを取るべきです：

### 1. 完成の定義の目的と重要性の説明

**完成の定義の目的**
- 完成の定義は、インクリメントが「完成」したことを示す基準であり、品質の保証です
- 完成の定義により、チームは一貫した品質基準を維持し、透明性を確保します
- 完成の定義を満たしたインクリメントのみが、スプリントレビューで提示され、価値を提供できます

**完成の定義を緩めることの問題**
- 完成の定義を緩めることは、品質基準を下げることであり、技術的な負債を蓄積させます
- 完成の定義を緩めると、短期的には作業を完了できますが、長期的には品質と価値提供能力が低下します
- 完成の定義を緩めることで、透明性が失われ、ステークホルダーの信頼が損なわれる可能性があります

### 2. スキル不足への代替的な対応

**スキルの獲得**
- 欠けているスキルをチーム内で獲得する方法を検討します
- トレーニング、メンタリング、知識共有セッションなどを通じて、必要なスキルを開発します
- 他のチームメンバーが、欠けているスキルを学習するよう支援します

**ペアプログラミングとモブプログラミング**
- ペアプログラミングやモブプログラミングを通じて、スキルを共有します
- 経験のあるメンバーと、スキルを学ぶ必要があるメンバーをペアにします
- これにより、知識の共有とスキルの開発が同時に行われます

**外部リソースの活用**
- 組織内の他のチームや専門家から支援を得ることを検討します
- 一時的な外部リソースやコンサルタントを活用することができます
- ただし、長期的にはチーム内でスキルを開発することを目指します

### 3. スプリントプランニングの調整

**現実的なコミットメント**
- 欠けているスキルを考慮して、現実的なコミットメントを行うよう促します
- スプリントプランニングで、必要なスキルと利用可能なスキルを評価します
- スキル開発の時間を考慮した計画を立てます

**スキル開発の優先順位付け**
- スプリントバックログに、スキル開発のためのタスクを含めることを検討します
- ただし、スキル開発が過剰にならないよう、バランスを取ります
- 長期的なスキル開発計画を立てます

### 4. 透明性の確保

**状況の透明化**
- スキル不足の状況を、プロダクトオーナーやステークホルダーに透明に伝えます
- 完成の定義を満たすために必要なスキルと、現在の状況を説明します
- スキル不足が、スプリントゴールやインクリメントの提供に与える影響を説明します

**影響の可視化**
- 完成の定義を緩めた場合の長期的な影響を可視化します
- 技術的な負債の蓄積、品質の低下、将来の価値提供能力への影響を説明します

### 5. 完成の定義の段階的な改善（慎重に）

**段階的なアプローチ（例外の場合）**
- 完全に完成の定義を緩めるのではなく、段階的に改善することを検討します
- ただし、これは長期的な解決策ではなく、一時的な措置であることを明確にします
- 完成の定義を緩める場合でも、品質基準を大幅に下げるべきではありません

**目標の明確化**
- 完成の定義を一時的に調整する場合、元の基準に戻すための明確な計画を立てます
- スキル開発の目標とタイムラインを設定します
- 定期的に進捗を確認し、完成の定義を元の基準に戻すことを目指します

### 6. 組織的な支援の確保

**スキル開発への投資**
- 組織に対して、チームのスキル開発への投資を要求します
- トレーニング、メンタリング、知識共有のための時間とリソースを確保します
- 組織全体で、スキル開発の重要性を認識します

**リソースの確保**
- 必要に応じて、一時的な外部リソースや専門家の支援を組織に要求します
- 長期的には、チーム内でスキルを開発できるよう、組織的な支援を確保します

### 7. 継続的な改善

**スプリントレトロスペクティブでの議論**
- この状況をスプリントレトロスペクティブで議論します
- スキル開発の進捗と、完成の定義を満たすための方法を検討します
- 継続的にスキルを開発し、完成の定義を満たすための方法を改善します

**学習の文化**
- チームに学習の文化を築きます
- スキル開発を継続的な活動として取り入れます
- 知識共有とスキル開発を促進します

### 8. 推奨されるアプローチ

**完成の定義は維持する（推奨）**
- 完成の定義を緩めるのではなく、スキル不足を解決する方法を検討します
- トレーニング、メンタリング、知識共有を通じて、必要なスキルを開発します
- 透明性を保ちながら、段階的にスキルを開発します

**代替的な解決策**
- ペアプログラミング、外部リソースの活用、スプリントプランニングの調整など、完成の定義を緩める以外の方法を検討します
- 長期的な視点で、チームのスキルを向上させます

**透明性と継続的な改善**
- 状況を透明にし、ステークホルダーと協力して解決策を見出します
- 継続的にスキルを開発し、完成の定義を満たすことを目指します

### 重要な原則

- **完成の定義を緩めるのではなく、スキル不足を解決する**
- **品質基準を維持し、技術的な負債を蓄積させない**
- **透明性を保ち、段階的にスキルを開発する**
- **長期的な視点で、チームの能力を向上させる**

## Key Points
- 完成の定義の目的と重要性の説明（完成の定義の目的、完成の定義を緩めることの問題）
- スキル不足への代替的な対応（スキルの獲得、ペアプログラミングとモブプログラミング、外部リソースの活用）
- スプリントプランニングの調整（現実的なコミットメント、スキル開発の優先順位付け）
- 透明性の確保（状況の透明化、影響の可視化）
- 完成の定義の段階的な改善（慎重に）（段階的なアプローチ、目標の明確化）
- 組織的な支援の確保（スキル開発への投資、リソースの確保）
- 継続的な改善（スプリントレトロスペクティブでの議論、学習の文化）
- 完成の定義は維持し、スキル不足を解決する
- 透明性と継続的な改善の重要性
- 長期的な視点でのチーム能力の向上
