# Question 020: スプリントレビュー直前のプロセス逸脱と製品品質の問題発見

## Scenario
スプリントレビューの直前に、開発チームメンバーの一人が、プロセスの一部が許容範囲を超えており、結果として製品が受け入れられないものになることを特定しました。

## Question
スクラムマスターとして、この状況にどのように対応すべきでしょうか？

## Evaluation Criteria
1. **スクラムガイドの理解** - 透明性の重要性、スプリントレビューの目的、受け入れられない製品の提示に関する理解
2. **実践的な適用** - スプリントレビュー直前の発見に対する具体的な対応方法
3. **透明性と信頼** - 問題を隠すことなく、透明に共有することの重要性の理解
4. **学習目標達成** - 問題の学習目標に対する回答の完全性と明確性

## Model Answer

スクラムマスターとして、この状況は透明性と品質に関する重要な問題です。スプリントレビュー直前の発見であっても、問題を隠すことなく、透明に共有することが重要です。以下のアプローチを取るべきです：

### 1. 即座の対応と透明性の確保

**問題の確認と理解**
- 開発チームメンバーと共に、問題の詳細を確認します
- プロセスのどの部分が許容範囲を超えているのか、製品のどの部分が受け入れられないのかを明確にします
- 問題の深刻度と影響を評価します

**チーム全体での共有**
- 問題をチーム全体で即座に共有します
- 透明性を保ち、すべてのチームメンバーが状況を理解できるようにします
- 問題を隠すのではなく、オープンに議論します

### 2. スプリントレビューでの透明な報告

**問題の透明な提示**
- スプリントレビューでは、発見された問題を透明に提示します
- プロセスの逸脱と、それが製品に与えた影響を正直に報告します
- 受け入れられない製品であることを隠さず、状況を説明します

**ステークホルダーへの説明**
- ステークホルダーに対して、問題の内容と影響を明確に説明します
- なぜこの問題が発生したのか、どのような影響があるのかを説明します
- 今後の対応計画を説明します

### 3. スプリントレビューの目的の再確認

**スプリントレビューの本質**
- スプリントレビューは、成功した成果だけを報告する場ではありません
- スプリントレビューは、インクリメントを検査し、プロダクトバックログを適応させる機会です
- 問題を発見し、適応することは、スクラムの経験主義の実現です

**透明性の重要性**
- 問題を隠すことは、透明性を損ない、経験主義の実現を阻害します
- 透明性を保つことで、ステークホルダーと協力して問題を解決できます
- 透明性により、信頼関係を維持し、長期的な成功につながります

### 4. プロセス改善の検討

**根本原因の分析**
- なぜプロセスが許容範囲を超えたのか、その根本原因を分析します
- プロセスの問題、計画の問題、リソースの問題、その他の要因を特定します
- 問題が早期に発見されなかった理由を検討します

**改善策の検討**
- スプリントレトロスペクティブで、問題の根本原因と改善策を検討します
- プロセスの改善、計画の改善、早期発見のメカニズムの導入などを検討します
- このような問題が再発しないよう、プロセスを改善します

### 5. 受け入れられない製品の扱い

**完成の定義の確認**
- 受け入れられない製品は、完成の定義を満たしていません
- 完成の定義を満たしていないインクリメントは、スプリントレビューで提示する際に、その状況を明確に説明します
- 完成の定義が適切に適用されていたかを確認します

**次回のスプリントでの対応**
- 受け入れられない製品を修正するための作業を、次のスプリントで計画します
- プロダクトバックログに、修正のためのアイテムを追加するようプロダクトオーナーに促します
- 優先順位を評価し、適切に対応します

### 6. 早期発見のメカニズムの改善

**検査の頻度と質の向上**
- 問題が早期に発見されるよう、検査の頻度と質を向上させます
- デイリースクラムや継続的な検査を通じて、問題を早期に発見できるようにします
- 完成の定義を満たしているかを、定期的に確認します

**透明性とフィードバックループ**
- 問題を早期に発見できるよう、透明性とフィードバックループを改善します
- チームメンバーが問題を報告しやすい環境を作ります
- 問題を報告することに対して、否定的ではなく、建設的な態度を示します

### 7. チームへの支援とコーチング

**問題報告の促進**
- チームメンバーが問題を早期に報告できるよう支援します
- 問題を報告することの重要性を説明します
- 問題を隠すことのリスクを説明します

**透明性の文化**
- チームに透明性の文化を築きます
- 問題を隠すのではなく、オープンに議論する文化を促進します
- 失敗を学習の機会として捉える文化を作ります

### 8. 推奨されるアプローチ

**透明性を保つ（推奨）**
- 問題を隠すことなく、透明に共有します
- スプリントレビューで、問題を正直に報告します
- ステークホルダーと協力して、問題を解決します

**学習の機会として捉える**
- この問題を学習の機会として捉えます
- 根本原因を分析し、プロセスを改善します
- このような問題が再発しないよう、継続的に改善します

**信頼関係の維持**
- 透明性を保つことで、ステークホルダーとの信頼関係を維持します
- 問題を隠すことは、短期的には良いように見えますが、長期的には信頼を損ないます
- 正直さと透明性により、長期的な成功につながります

### 重要な原則

- **透明性を保ち、問題を隠すことは推奨されない**
- **スプリントレビューでは、問題を正直に報告し、適応する**
- **問題を学習の機会として捉え、プロセスを改善する**
- **早期発見のメカニズムを改善し、問題を早期に発見できるようにする**

## Key Points
- 即座の対応と透明性の確保（問題の確認と理解、チーム全体での共有）
- スプリントレビューでの透明な報告（問題の透明な提示、ステークホルダーへの説明）
- スプリントレビューの目的の再確認（スプリントレビューの本質、透明性の重要性）
- プロセス改善の検討（根本原因の分析、改善策の検討）
- 受け入れられない製品の扱い（完成の定義の確認、次回のスプリントでの対応）
- 早期発見のメカニズムの改善（検査の頻度と質の向上、透明性とフィードバックループ）
- チームへの支援とコーチング（問題報告の促進、透明性の文化）
- 透明性を保ち、学習の機会として捉える
- 信頼関係の維持の重要性
- 早期発見と継続的な改善
