# Question 021: スプリントレビュー中の完成の定義に関する意見の不一致

## Scenario
スプリントレビュー中、開発チームがインクリメントについて質問に答えている際、特定のプロダクトバックログアイテムが「完成」と見なされるかどうかについて議論が生じました。一部の開発チームメンバーは「完成」と主張し、他のメンバーは「完成」でないと主張しています。

## Question
スクラムマスターとして、この状況にどのように対応すべきでしょうか？

## Evaluation Criteria
1. **スクラムガイドの理解** - 完成の定義の重要性と、チーム内の意見の不一致の対処方法の理解
2. **実践的な適用** - スプリントレビュー中の意見の不一致に対する具体的な対応方法
3. **チームの合意形成** - 完成の定義の明確化とチーム内の合意形成プロセス
4. **学習目標達成** - 問題の学習目標に対する回答の完全性と明確性

## Model Answer

スクラムマスターとして、この状況は完成の定義の重要性と、チーム内の合意形成に関する重要な問題です。スプリントレビュー中に意見の不一致が表面化したことは、完成の定義が明確でない、またはチーム全体で合意されていないことを示しています。以下のアプローチを取るべきです：

### 1. スプリントレビュー中の対応

**状況の整理とファシリテーション**
- スプリントレビューの場で、議論を整理し、ファシリテーションを行います
- 両方の立場を理解し、意見の不一致の原因を特定します
- 完成の定義を確認し、そのアイテムが定義を満たしているかどうかを評価します

**透明性の確保**
- 意見の不一致を隠すのではなく、透明に扱います
- ステークホルダーに対して、現在の状況を説明します
- 完成の定義に関する意見の不一致があり、チーム内で検討が必要であることを説明します

**一時的な対応**
- スプリントレビュー中は、意見の不一致を認めつつ、そのアイテムの状態を透明に説明します
- 完成の定義を確認し、チーム内で合意を形成する必要があることを説明します
- 詳細な議論は、スプリントレビュー後にチーム内で行うことを提案します

### 2. 完成の定義の確認と明確化

**完成の定義の確認**
- 開発チームの完成の定義を確認します
- 完成の定義が文書化され、チーム全体で共有されているかを確認します
- 完成の定義が明確で、測定可能であるかを評価します

**意見の不一致の原因の特定**
- なぜ意見が分かれているのか、その原因を特定します
  - 完成の定義が不明確である
  - 完成の定義がチーム全体で合意されていない
  - 完成の定義の解釈が異なる
  - アイテムの特定の側面が完成の定義でカバーされていない
- 意見の不一致の具体的なギャップや解釈の違いを特定します

### 3. チーム内での合意形成

**チーム内での対話の促進**
- スプリントレビュー後に、チーム全体で対話を促進します
- 両方の立場を尊重し、建設的な議論を促進します
- 完成の定義を確認し、そのアイテムが定義を満たしているかどうかを評価します

**合意形成プロセス**
- チーム全体で、完成の定義を再確認し、合意を形成します
- 意見の不一致を解決するために、完成の定義を明確にし、解釈を統一します
- チーム全体で、そのアイテムの状態について合意を形成します

**完成の定義の更新**
- 必要に応じて、完成の定義を更新します
- 完成の定義を明確にし、曖昧さを排除します
- 完成の定義を文書化し、チーム全体で共有します

### 4. 完成の定義の改善

**明確性の向上**
- 完成の定義が明確で、測定可能であることを確保します
- 完成の定義が、すべてのチームメンバーが同じように理解できるよう、明確に定義します
- 完成の定義に、具体的な基準とチェックリストを含めます

**チーム全体での合意**
- 完成の定義が、チーム全体で合意されていることを確保します
- 新しいメンバーが加わった場合、完成の定義を説明し、合意を得ます
- 完成の定義に関する定期的なレビューを行います

**継続的な改善**
- 完成の定義を、スプリントレトロスペクティブで定期的にレビューします
- 意見の不一致や問題が発生した場合、完成の定義を改善します
- 完成の定義を継続的に改善し、明確性と実用性を向上させます

### 5. 透明性と信頼関係の維持

**ステークホルダーへの透明な説明**
- ステークホルダーに対して、意見の不一致とその解決プロセスを透明に説明します
- 完成の定義を明確にし、将来このような問題が発生しないようにする計画を説明します
- 透明性を保ち、信頼関係を維持します

**学習の機会として捉える**
- この意見の不一致を学習の機会として捉えます
- 完成の定義を改善し、チーム全体で合意を形成することで、将来このような問題を防ぎます
- この経験から学び、完成の定義の重要性を再認識します

### 6. プロセスの改善

**早期発見のメカニズム**
- 完成の定義に関する意見の不一致を、スプリントレビューまで発見できなかった理由を検討します
- デイリースクラムや継続的な検査を通じて、早期に意見の不一致を発見できるようにします
- 完成の定義を満たしているかを、定期的に確認するプロセスを確立します

**合意形成のプロセス**
- スプリントプランニングやリファインメントセッションで、完成の定義を確認するプロセスを確立します
- アイテムが完成の定義を満たしているかを、チーム全体で確認するプロセスを確立します
- 意見の不一致が発生した場合、早期に解決するプロセスを確立します

### 7. スクラムマスターの役割

**ファシリテーション**
- チーム内での対話を促進し、合意形成を支援します
- 完成の定義に関する議論をファシリテーションし、建設的な解決策を見出します
- チームが自分たちで問題を解決できるよう支援します

**教育とコーチング**
- チームに対して、完成の定義の重要性を教育します
- 完成の定義が明確で、チーム全体で合意されていることの重要性を説明します
- チームが完成の定義を適切に使用できるようコーチングします

### 8. 推奨されるアプローチ

**即座の対応**
- スプリントレビュー中は、意見の不一致を認め、透明に説明します
- 詳細な議論は、スプリントレビュー後にチーム内で行うことを提案します
- ステークホルダーに対して、状況と対応計画を説明します

**根本的な解決**
- スプリントレビュー後に、チーム全体で完成の定義を再確認し、合意を形成します
- 必要に応じて、完成の定義を更新し、明確にします
- このような問題が再発しないよう、プロセスを改善します

**継続的な改善**
- 完成の定義を定期的にレビューし、改善します
- 意見の不一致を早期に発見し、解決するプロセスを確立します
- チーム全体で、完成の定義の重要性を再認識します

### 重要な原則

- **完成の定義は、チーム全体で明確に定義され、合意される必要がある**
- **意見の不一致を透明に扱い、チーム内で合意を形成する**
- **完成の定義を継続的に改善し、明確性を向上させる**
- **意見の不一致を学習の機会として捉え、プロセスを改善する**

## Key Points
- スプリントレビュー中の対応（状況の整理とファシリテーション、透明性の確保、一時的な対応）
- 完成の定義の確認と明確化（完成の定義の確認、意見の不一致の原因の特定）
- チーム内での合意形成（チーム内での対話の促進、合意形成プロセス、完成の定義の更新）
- 完成の定義の改善（明確性の向上、チーム全体での合意、継続的な改善）
- 透明性と信頼関係の維持（ステークホルダーへの透明な説明、学習の機会として捉える）
- プロセスの改善（早期発見のメカニズム、合意形成のプロセス）
- スクラムマスターの役割（ファシリテーション、教育とコーチング）
- 即座の対応と根本的な解決
- 継続的な改善の重要性
- 完成の定義の明確化とチーム全体での合意

## Source

The Development Team members that worked on the item are in disagreement. Some argue it is "Done", others argue it is not.

## Notes
この問題では、スクラムマスターが、スプリントレビュー中の意見の不一致を透明に扱い、チーム内で合意を形成することが重要です。完成の定義が明確で、チーム全体で合意されていることが重要であり、意見の不一致が発生した場合は、完成の定義を再確認し、明確化する必要があります。このような問題を学習の機会として捉え、完成の定義を改善し、プロセスを継続的に改善することが求められます。
