# Question 016: スプリント中のアイテム変更（複雑さの見積もり誤差）

## Scenario
スプリント中に、開発チームが予測で選択したアイテムが、見積もりよりもはるかに複雑であることが判明しました。プロダクトオーナーの承認を得て、開発チームは完了できると考える別のアイテムに変更しました。

## Question
スクラムマスターとして、この状況にどのように対応すべきでしょうか？

## Evaluation Criteria
1. **スクラムガイドの理解** - スプリント中のスコープ変更、スプリントゴールの保護、透明性の重要性の理解
2. **実践的な適用** - 見積もりの不確実性とスプリント中の適応に対する具体的な対応方法
3. **プロダクトオーナーと開発チームの協力** - 適切な対話と意思決定プロセスの理解
4. **学習目標達成** - 問題の学習目標に対する回答の完全性と明確性

## Model Answer

スクラムマスターとして、この状況はスプリント中の適応とスプリントゴールの保護に関する重要な問題です。プロダクトオーナーの承認を得た変更は適切な対応ですが、いくつかの重要な観点があります。以下のアプローチを取るべきです：

### 1. 状況の理解と評価

**変更の適切性の評価**
- プロダクトオーナーの承認を得て変更したことは、透明性と協力を示しています
- スプリント中に見積もりの誤差が判明した場合、適応することはスクラムの原則に合致します
- ただし、変更がスプリントゴールに与える影響を評価する必要があります

**スプリントゴールへの影響**
- 変更したアイテムがスプリントゴールに関連しているかを確認します
- スプリントゴールが達成できるかを評価します
- スプリントゴールが変更された場合、その影響を透明にします

### 2. スプリントゴールの保護

**スプリントゴールの重要性**
- スプリントゴールは、スプリント中に変更されない目標であり、チームの焦点を維持するために重要です
- アイテムを変更する場合でも、スプリントゴールが達成できることを確保する必要があります
- スプリントゴールが変更された場合、その理由と影響を透明にします

**スプリントゴールへの影響の最小化**
- 可能であれば、スプリントゴールを維持しつつ、アイテムを変更することを推奨します
- 変更したアイテムが、スプリントゴールの達成に貢献することを確認します

### 3. 透明性の確保

**変更の透明化**
- アイテムの変更とその理由を、チーム全体で透明にします
- スプリントレビューで、変更の理由と影響をステークホルダーに説明できるよう準備します
- 変更がスプリントゴールや他のアイテムに与える影響を透明にします

**ステークホルダーへの説明**
- 必要に応じて、ステークホルダーに対して変更の理由を説明します
- 見積もりの不確実性と、適応の必要性を説明します

### 4. 見積もりの改善

**見積もりの精度向上**
- 見積もりの誤差が発生した理由を分析します
- プロダクトバックログリファインメントの質を向上させる必要性を検討します
- リファインメントセッションで、アイテムの複雑さをより適切に評価する方法を改善します

**不確実性の管理**
- 見積もりには常に不確実性があることを認識します
- 不確実性が大きいアイテムについては、バッファや余裕を考慮した計画を立てるよう促します
- スプリント中に見積もりの誤差が判明することは、経験主義の実現（検査と適応）の一部です

### 5. スプリントプランニングの改善

**プランニングの質の向上**
- スプリントプランニングで、アイテムの複雑さをより適切に評価する方法を改善します
- 技術的なリスクや不確実性を早期に特定するよう促します
- 開発チームが、アイテムの複雑さをより正確に評価できるよう支援します

**プロダクトバックログリファインメントの強化**
- リファインメントセッションで、アイテムの詳細を十分に議論するよう促します
- 技術的な調査やスパイクを検討し、不確実性を減らします
- アイテムが「Ready」状態になる前に、複雑さを適切に評価します

### 6. 経験主義の実現

**検査と適応**
- 見積もりの誤差が判明したことは、経験主義の実現（検査）です
- プロダクトオーナーの承認を得て変更したことは、適応の実践です
- 透明性を保ちながら、適切に適応することが重要です

**学習の機会**
- この状況を学習の機会として捉えます
- 見積もりの精度を向上させる方法を検討します
- スプリントレトロスペクティブで、この経験から学んだことを議論します

### 7. スクラムマスターの役割

**チームへの支援**
- 開発チームが、見積もりの誤差を適切に処理できるよう支援します
- プロダクトオーナーとの対話を促進し、適切な判断ができるよう支援します
- チームが経験主義の原則（透明性、検査、適応）に従って行動しているかを確認します

**プロセスの改善**
- 見積もりの誤差を減らすためのプロセスの改善を支援します
- リファインメントセッションの質を向上させるよう促します
- スプリントプランニングの精度向上を支援します

### 8. 推奨されるアプローチ

**透明性と適応の支援（推奨）**
- プロダクトオーナーの承認を得て変更したことは適切な対応です
- 変更の理由と影響を透明にし、スプリントゴールへの影響を評価します
- 見積もりの改善に向けて、学習の機会として活用します

**スプリントゴールの保護**
- 可能であれば、スプリントゴールを維持しつつアイテムを変更します
- スプリントゴールが変更された場合、その理由と影響を透明にします

**継続的な改善**
- この経験から学び、見積もりの精度を向上させます
- リファインメントとプランニングの質を向上させます

### 重要な原則

- **スプリント中に見積もりの誤差が判明した場合、適応することは適切**
- **プロダクトオーナーの承認を得た変更は、透明性と協力を示している**
- **スプリントゴールへの影響を評価し、透明性を保つ**
- **見積もりの誤差を学習の機会として捉え、継続的に改善する**

## Key Points
- 状況の理解と評価（変更の適切性の評価、スプリントゴールへの影響）
- スプリントゴールの保護（重要性、影響の最小化）
- 透明性の確保（変更の透明化、ステークホルダーへの説明）
- 見積もりの改善（精度向上、不確実性の管理）
- スプリントプランニングの改善（質の向上、プロダクトバックログリファインメントの強化）
- 経験主義の実現（検査と適応、学習の機会）
- スクラムマスターの役割（チームへの支援、プロセスの改善）
- 透明性と適応の支援（推奨）
- スプリントゴールの保護と継続的な改善
- 見積もりの誤差を学習の機会として捉える
